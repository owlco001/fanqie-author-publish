# 测试用例

这个文件用于验证 `fanqie-author-publish` 在 Linux / OpenClaw 环境中的可执行性。

目标不是一次性覆盖所有细节，而是先覆盖最常见、最关键的成功路径和失败路径。

## 测试前提

执行测试前，默认满足以下条件：
- OpenClaw 已可调用浏览器工具
- 使用独立的作者账号浏览器 profile
- 技能目录已可读写
- 输入文件为 UTF-8 编码
- 日志目录和截图目录可写

建议测试顺序：
1. 单章草稿成功路径
2. 单章发布前检查路径
3. 单章发布确认路径
4. 批量草稿路径
5. 异常与停止路径

## 用例 1：单章草稿保存成功

- 编号：`TC-001`
- 类型：成功路径
- 输入：`chapter.example.json`
- 动作：`prepare_draft`

预期步骤：
- 打开番茄作者后台
- 定位目标作品
- 打开章节编辑器
- 填入标题和正文
- 点击保存草稿

预期结果：
- 输出 `state: draft_saved`
- `result` 明确说明草稿已保存
- 生成日志文件
- 至少生成 1 到 2 张关键截图

失败判定：
- 标题或正文未成功写入
- 保存草稿后没有任何成功信号

## 用例 2：发布前检查成功

- 编号：`TC-002`
- 类型：成功路径
- 输入：已存在草稿的单章数据
- 动作：`review`

预期步骤：
- 打开目标作品的章节编辑页或章节列表
- 检查标题、正文、草稿状态、发布按钮状态

预期结果：
- 输出 `ready to publish` 或等价成功描述
- `state` 不应是失败态
- 如需人工确认，`confirmation_required` 为 `true`

失败判定：
- 无法确认作品或章节
- 页面上缺少关键字段导致无法判断是否可发布

## 用例 3：单章发布成功

- 编号：`TC-003`
- 类型：成功路径
- 输入：已准备好草稿的单章数据
- 动作：`publish`

前置条件：
- 用户已在当前会话明确确认发布

预期步骤：
- 再次确认作品名和章节标题
- 点击发布
- 如有弹窗，只确认一次
- 回到章节列表或看到成功提示

预期结果：
- 输出 `state: published_verified`
- `result` 明确说明已发布
- 日志文件中包含发布结果

失败判定：
- 重复点击发布
- 发布后无成功信号，且仍被错误标记为已发布

## 用例 4：批量草稿准备成功

- 编号：`TC-004`
- 类型：成功路径
- 输入：`chapters.batch.example.json`
- 动作：`prepare_draft`

预期步骤：
- 按顺序逐章处理
- 每章都完成标题填写、正文填写、草稿保存

预期结果：
- 每章都有单独结果记录
- 出现失败前，不应跳过未处理章节
- 总处理数量不超过 schema 限制

失败判定：
- 批量顺序错乱
- 某章失败后仍无说明地继续执行后续章节

## 用例 5：未登录后台

- 编号：`TC-005`
- 类型：失败路径
- 输入：任意合法单章输入
- 动作：`prepare_draft`

场景：
- 打开的是登录页，未处于作者后台登录状态

预期结果：
- 立即停止
- 输出 `state: failed_needs_review`
- `blocking_reason` 指向登录失效或未登录
- `requires_manual_action` 为 `true`

失败判定：
- 在登录页继续尝试查找作品或输入内容

## 用例 6：出现验证码或风控

- 编号：`TC-006`
- 类型：失败路径
- 输入：任意合法输入
- 动作：任意

场景：
- 页面出现验证码、身份校验、风控提示

预期结果：
- 立即停止
- 输出 `state: failed_needs_review`
- 日志中记录验证码或风控
- 不继续任何点击

失败判定：
- 继续尝试自动处理验证码
- 在风控页面上继续操作

## 用例 7：找不到目标作品

- 编号：`TC-007`
- 类型：失败路径
- 输入：不存在的 `book_name`
- 动作：`prepare_draft`

预期结果：
- 停止在作品定位阶段
- 输出明确错误：未找到目标作品
- 记录最后成功步骤为后台已打开或作品列表已打开

失败判定：
- 在未确认作品时误点其他作品进入

## 用例 8：章节标题重复

- 编号：`TC-008`
- 类型：失败路径
- 输入：与平台已有章节同名的标题
- 动作：`prepare_draft` 或 `publish`

预期结果：
- 停止并报告标题重复
- 默认不继续覆盖或重复发布
- `requires_manual_action` 为 `true`

失败判定：
- 未经确认直接覆盖已有内容
- 在重复标题下继续发布

## 用例 9：编辑器未接受正文

- 编号：`TC-009`
- 类型：失败路径
- 输入：合法单章数据
- 动作：`prepare_draft`

场景：
- 标题已输入，但正文编辑器未显示内容或仅显示部分内容

预期结果：
- 停止在正文写入阶段
- 输出 `state: failed_needs_review`
- 截图保留编辑器当前状态

失败判定：
- 未确认正文是否存在就继续保存草稿

## 用例 10：保存草稿结果不明确

- 编号：`TC-010`
- 类型：失败路径
- 输入：合法单章数据
- 动作：`prepare_draft`

场景：
- 点击保存草稿后没有 toast，也没有明确状态变化

预期结果：
- 不重复点击保存草稿
- 尝试通过章节列表或页面状态确认
- 若仍不明确，则停止并报告结果不明确

失败判定：
- 连续多次点击保存草稿

## 用例 11：发布结果不明确

- 编号：`TC-011`
- 类型：失败路径
- 输入：合法单章数据
- 动作：`publish`

场景：
- 点击发布后没有成功提示，也没有明显跳转或状态变化

预期结果：
- 不再次点击发布
- 优先去章节列表检查状态
- 若仍无法确认，停止并标记人工复查

失败判定：
- 为了“确认一下”再次点击发布

## 用例 12：批量处理中途失败

- 编号：`TC-012`
- 类型：失败路径
- 输入：批量章节数据
- 动作：`prepare_draft`

场景：
- 第 2 章处理时出现错误

预期结果：
- 默认停止后续章节
- 输出当前失败章节和阻塞原因
- 保留前面已成功章节的结果记录

失败判定：
- 未说明原因就继续处理第 3 章之后的内容

## 最低验收标准

至少满足以下条件，才能认为第一版技能包可在 Linux / OpenClaw 环境中试运行：

- `TC-001` 通过
- `TC-002` 通过
- `TC-003` 在人工确认场景下通过
- `TC-005`、`TC-006`、`TC-009`、`TC-011` 能正确停止
- 日志和截图路径可正常生成
- 输入输出均能通过 schema 校验

## 建议记录方式

每次跑测试时，建议补充：
- 测试时间
- 使用的输入文件
- 实际结果
- 是否符合预期
- 失败截图路径
- 是否需要调整 `selectors.md` 或 `execution-policy.md`

这样后面可以持续迭代这套技能包，而不是每次从头排查。
